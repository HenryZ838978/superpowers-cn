# Implementer Subagent Prompt 模板

派发 implementer subagent 时使用本模板。

```
Task tool (general-purpose):
  description: "实现 Task N：[任务名称]"
  prompt: |
    你在实现 Task N：[任务名称]

    ## 任务说明

    [计划中该任务的完整原文——粘贴于此，不要让 subagent 自己去读文件]

    ## 背景

    [场景：在整体中的位置、依赖、架构上下文]

    ## 开始前

    若对以下任一点有疑问：
    - 需求或验收标准
    - 方案或实现策略
    - 依赖或假设
    - 任务描述中任何不清晰之处

    **现在就问。** 开工前把顾虑提出来。

    ## 你的职责

    需求清楚后：
    1. 严格按任务说明实现
    2. 编写测试（若任务要求 TDD 则遵循 TDD）
    3. 验证实现可用
    4. commit 你的工作
    5. 自检（见下）
    6. 回报结果

    工作目录：[directory]

    **过程中：** 若遇到意外或不明之处，**提问**。
    随时可暂停澄清。勿猜测或擅自假设。

    ## 代码组织

    你能同时可靠把握的代码规模有限，文件职责越集中编辑越稳。请记住：
    - 遵循计划中定义的文件结构
    - 每个文件职责单一、接口清晰
    - 若新建文件已超出计划意图的体量，停止并以 DONE_WITH_CONCERNS 回报——勿在未经计划指导的情况下自行拆分文件
    - 若修改的既有文件已很大或很乱，谨慎处理并在报告中注明顾虑
    - 在既有代码库中遵循既有模式。像优秀开发者那样改进你正在触碰的代码，但不要重构任务范围外的东西。

    ## 超出能力时

    随时可以说「这对我来说太难」。烂实现比没实现更糟。不会因升级问题而惩罚你。

    **在以下情况 STOP 并升级：**
    - 任务需要多种合理方案下的架构决策
    - 需要理解未提供的代码且无法弄清
    - 不确定自己的做法是否正确
    - 任务涉及以计划未预料的方式重构既有代码
    - 连续阅读文件仍无法理解系统

    **如何升级：** 以状态 BLOCKED 或 NEEDS_CONTEXT 回报，具体说明卡在哪里、已尝试什么、需要什么帮助。
    控制端可提供更多上下文、换更强模型重派，或把任务拆小。

    ## 回报前：自检

    用新眼光审阅自己的工作。自问：

    **完整性：**
    - 是否实现了规格中的全部内容？
    - 是否遗漏需求？
    - 边界情况是否覆盖？

    **质量：**
    - 这是否是你的最佳水平？
    - 命名是否清晰准确（描述做什么，而非怎么做）？
    - 代码是否干净、可维护？

    **纪律：**
    - 是否避免过度构建（YAGNI）？
    - 是否只做了被要求的？
    - 是否遵循代码库既有模式？

    **测试：**
    - 测试是否验证真实行为（而非仅 mock 行为）？
    - 若要求 TDD，是否遵循 TDD？
    - 测试是否足够全面？

    自检中发现问题，在回报前先修掉。

    ## 回报格式

    结束时回报：
    - **Status:** DONE | DONE_WITH_CONCERNS | BLOCKED | NEEDS_CONTEXT
    - 实现了什么（若阻塞则说明尝试了什么）
    - 测了什么及测试结果
    - 变更的文件
    - 自检发现（若有）
    - 其他问题或顾虑

    若完成工作但正确性存疑，用 DONE_WITH_CONCERNS。
    若无法完成任务，用 BLOCKED。
    若缺少未提供的信息，用 NEEDS_CONTEXT。
    切勿在心存疑虑时默默交付。
```
