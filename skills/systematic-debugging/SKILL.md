# 系统化调试

## 概述

随机打补丁浪费时间，还会制造新 bug。快速掩盖无法解决根因。

**核心原则：** 在尝试修复**之前**必须找到根因。只治症状等于失败。

**违反本流程的字面规定，就是违反调试的精神。**

## 铁律

```
NO FIXES WITHOUT ROOT CAUSE INVESTIGATION FIRST
```

若未完成阶段 1，不得提出修复方案。

## 何时使用

适用于**任何**技术问题：
- 测试失败  
- 生产环境 bug  
- 异常行为  
- 性能问题  
- 构建失败  
- 集成问题  

**尤其在以下情况使用：**
- 时间紧迫（紧急时更容易想猜）
- 「就改这一处」看起来很明显
- 已经试过多种修复
- 上一版修复无效
- 尚未完全理解问题  

**不要因以下理由跳过：**
- 问题看起来简单（简单 bug 也有根因）
- 你很赶（匆忙必然返工）
- 经理要求马上修好（系统化比乱试更快）

## 四个阶段

进入下一阶段**之前**必须完成当前阶段。

### 阶段 1：根因调查

**在尝试任何修复之前：**

1. **仔细阅读错误信息**
   - 不要跳过错误或警告
   - 答案经常在错误里
   - 完整阅读堆栈
   - 记下行号、文件路径、错误码

2. **稳定复现**
   - 能否可靠触发？
   - 具体步骤是什么？
   - 是否每次必现？
   - 若无法复现 → 多收集数据，不要猜

3. **检查近期变更**
   - 什么改动可能导致？
   - Git diff、近期 commit
   - 新依赖、配置变化
   - 环境差异

4. **在多组件系统中收集证据**

   **当系统含多个组件（CI → build → 签名，API → 服务 → 数据库）时：**

   **在提出修复前，先加诊断埋点：**
   ```
   For EACH component boundary:
     - Log what data enters component
     - Log what data exits component
     - Verify environment/config propagation
     - Check state at each layer

   Run once to gather evidence showing WHERE it breaks
   THEN analyze evidence to identify failing component
   THEN investigate that specific component
   ```

   **示例（多层系统）：**
   ```bash
   # Layer 1: Workflow
   echo "=== Secrets available in workflow: ==="
   echo "IDENTITY: ${IDENTITY:+SET}${IDENTITY:-UNSET}"

   # Layer 2: Build script
   echo "=== Env vars in build script: ==="
   env | grep IDENTITY || echo "IDENTITY not in environment"

   # Layer 3: Signing script
   echo "=== Keychain state: ==="
   security list-keychains
   security find-identity -v

   # Layer 4: Actual signing
   codesign --sign "$IDENTITY" --verbose=4 "$APP"
   ```

   **由此可知：** 哪一层失败（secrets → workflow ✓，workflow → build ✗）

5. **追踪数据流**

   **当错误深埋在调用栈中时：**

   见本目录 `root-cause-tracing.md` 了解完整反向追踪技术。

   **简版：**
   - 错误值从哪来？
   - 谁用错误值调用了这里？
   - 一路向上追到源头
   - 在源头修，不在症状处修

### 阶段 2：模式分析

**修复前先找模式：**

1. **找正常工作的例子**
   - 在同一代码库找相似且正常的代码
   - 什么能工作、与坏的有何相似？

2. **对照参考实现**
   - 若在实现某种模式，**完整**阅读参考实现
   - 不要扫读——逐行读
   - 应用前充分理解模式

3. **识别差异**
   - 正常与异常之间有什么不同？
   - 列出每一处差异，再小也别假设「无关紧要」

4. **理解依赖**
   - 还需要哪些组件？
   - 哪些设置、配置、环境？
   - 隐含了哪些假设？

### 阶段 3：假设与验证

**科学方法：**

1. **形成单一假设**
   - 明确写出：「我认为根因是 X，因为 Y」
   - 写下来
   - 要具体，不要模糊

2. **最小化验证**
   - 用**最小**变更验证假设
   - 一次只动一个变量
   - 不要同时改多处

3. **继续前先核实**
   - 有效？→ 阶段 4  
   - 无效？→ 形成**新**假设  
   - 不要在无效假设上再叠修复  

4. **不知道时**
   - 明说「我不懂 X」
   - 不要装懂
   - 求助或继续查资料

### 阶段 4：实现

**修根因，不治症状：**

1. **先写失败测试用例**
   - 最简单的复现
   - 尽量自动化测试
   - 若无框架可用一次性脚本
   - 修复**前**必须有
   - 写正规失败测试可用 `superpowers:test-driven-development` skill

2. **单一修复**
   - 针对已识别的根因
   - **一次**只改一处
   - 不要「顺手」改进
   - 不要捆绑重构

3. **验证修复**
   - 测试现在通过？
   - 其他测试仍通过？
   - 问题真解决了？

4. **若修复无效**
   - **停**
   - 计数：已尝试几种修复？
   - 若 < 3：回到阶段 1，用新信息再分析  
   - **若 ≥ 3：停，质疑架构（见下方第 5 步）**  
   - 未经架构讨论不要尝试第 4 种修复  

5. **若 3+ 次修复失败：质疑架构**

   **暗示架构问题的模式：**
   - 每次修复在不同位置暴露新的共享状态/耦合/问题  
   - 修复需要「大重构」才能实现  
   - 每次修复在别处制造新症状  

   **停，质疑基本假设：**
   - 这种模式根本上是否合理？
   - 是否在「凭惯性硬撑」？
   - 应重构架构还是继续治症状？

   **在尝试更多修复前与人类伙伴讨论**

   这不是失败的假设——这是错误的架构。

## 危险信号 — 停下并按流程来

若你发现自己这样想：
- 「先 quick fix，以后再查」
- 「先改 X 试试行不行」
- 「多处一起改，跑测试」
- 「跳过测试，我手动验」
- 「多半是 X，我改那个」
- 「没完全懂，但可能管用」
- 「文档说 X 但我换个写法」
- 「主要问题有这些：[列出修复而未调查]」
- 在追踪数据流之前就提方案
- **「再试一次修复」（已经试过 2+ 次）**
- **每次修复在不同地方冒新问题**

**以上任一条都表示：停。回到阶段 1。**

**若 3+ 次修复失败：** 质疑架构（见阶段 4.5）

## 人类伙伴提示「你做法不对」的信号

**注意这些纠正：**
- 「是不是其实没发生？」— 你在假设而未核实  
- 「能不能让我们看到…？」— 你本该先收集证据  
- 「别猜」— 未理解就提修复  
- 「Ultrathink this」— 质疑基本点，不仅是症状  
- 「我们是不是卡住了？」（沮丧）— 你的方法行不通  

**看到以上：** 停。回到阶段 1。

## 常见自我辩解

| 借口 | 现实 |
|--------|---------|
| 「问题简单，不用流程」 | 简单问题也有根因。流程对简单 bug 也很快。 |
| 「紧急，没时间走流程」 | 系统化调试比猜谜式乱试更快。 |
| 「先试试这个，再调查」 | 第一次修复定调性。一开始就做对。 |
| 「确认修复有效后再写测试」 | 无测试的修复站不住。先有测试才能证明。 |
| 「多处一起改省时间」 | 无法隔离哪条有效。还会引入新 bug。 |
| 「参考太长，我变通一下」 | 一知半解必出 bug。要完整读完。 |
| 「我看到问题了，我来修」 | 看到症状 ≠ 理解根因。 |
| 「再试一次修复」（已失败 2+ 次） | 3+ 失败 = 架构问题。质疑模式，别再瞎修。 |

## 速查

| 阶段 | 关键活动 | 成功标准 |
|-------|---------------|------------------|
| **1. 根因** | 读错误、复现、查变更、收集证据 | 理解**是什么**和**为什么** |
| **2. 模式** | 找正常例、对比 | 找出差异 |
| **3. 假设** | 立论、最小验证 | 证实或换新假设 |
| **4. 实现** | 写测试、修复、验证 | Bug 解决，测试通过 |

## 当流程显示「没有根因」

若系统化调查表明问题确属环境、时序或外部因素：

1. 你已走完流程  
2. 记录调查过程  
3. 实现适当处理（重试、超时、错误信息）  
4. 增加监控/日志供日后排查  

**但：** 声称「无根因」的案例里约 95% 是调查不彻底。

## 配套技术

下列技术属于系统化调试，见本目录：

- **`root-cause-tracing.md`** — 沿调用栈反向追踪，找到原始触发点  
- **`defense-in-depth.md`** — 找到根因后在多层加校验  
- **`condition-based-waiting.md`** — 用条件轮询替代随意 timeout  

**相关 skill：**
- **superpowers:test-driven-development** — 写失败测试用例（阶段 4 第 1 步）  
- **superpowers:verification-before-completion** — 宣称成功前先验证修复有效  

## 实际影响

来自调试会话：
- 系统化：约 15–30 分钟修好  
- 随机乱试：约 2–3 小时折腾  
- 一次修成率：约 95% vs 40%  
- 引入新 bug：几乎为零 vs 常见
